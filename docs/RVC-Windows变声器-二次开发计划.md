## 2. 二次开发目标

将当前 RVC WebUI/实时 GUI 工具二次开发为 Windows 桌面变声器软件，重点实现：

- 用户打开软件后可以选择真实麦克风、声音模型和输出设备。
- 用户边说话，软件实时完成 RVC 音色转换。
- 变声结果可以输出到虚拟麦克风，供 QQ、微信、Discord、游戏语音等软件使用。
- 软件界面从原始 WebUI/FreeSimpleGUI 形态升级为更美观的深色专业音频控制台。
- 第一版不做完整训练平台，优先保证实时变声、文件变声、模型管理和安装运行体验。

## 4. 推荐整体架构

```text
Windows 桌面客户端
  ↓
Tauri + React + TypeScript 前端
  ↓
本地 FastAPI 推理服务
  ↓
RVC 文件变声模块 / RVC 实时变声模块
  ↓
真实麦克风输入 / 音频文件输入
  ↓
扬声器监听 / 虚拟麦克风输出 / 文件导出
```

### 4.1 前端客户端

- 使用 Tauri 作为 Windows 桌面壳。
- 使用 React18 + TypeScript 实现界面。
- 使用 Zustand 保存运行状态。
- 使用 React Router 划分页面。
- 采用深色专业控制台风格，突出模型、设备、延迟、音量和运行状态。

### 4.2 后端服务

- 保留 Python 推理核心，不重写 RVC 模型逻辑。
- 基于 `api_240604.py` 拆分出更稳定的本地服务入口。
- 提供模型管理、设备检测、实时变声、文件变声、状态查询接口。
- 服务仅监听本机地址，避免局域网暴露造成滥用风险。

### 4.3 音频输出方案

- 第一版优先兼容现有虚拟声卡，例如 VB-CABLE 或 VoiceMeeter。
- 软件负责把变声后的声音输出到用户选择的虚拟输出设备。
- 用户在通话或游戏软件中选择该虚拟麦克风，即可让队友听到变声后的声音。
- 自研虚拟麦克风驱动不放入第一版，原因是涉及驱动签名、管理员权限和兼容性测试。

## 5. 功能开发计划

### 阶段一：项目封装和基础桌面端

- 新增 Tauri + React + TypeScript 桌面端工程。
- 设计深色主界面，包含左侧导航、中间控制台、右侧状态面板。
- 实现首页状态卡片：当前模型、麦克风、输出设备、GPU、延迟、运行状态。
- 实现本地配置保存：模型路径、输入设备、输出设备、参数预设。
- 保留原 RVC 目录结构，避免破坏 `infer-web.py` 和 `gui_v1.py` 的原始运行方式。

### 阶段二：本地推理服务整理

- 以 `api_240604.py` 为基础，新建面向桌面端的本地服务入口。
- 拆分设备查询接口：输入设备、输出设备、虚拟声卡候选设备。
- 拆分模型加载接口：加载 `.pth` 和 `.index`。
- 拆分状态查询接口：运行中、已停止、当前延迟、最后错误、设备状态。
- 增加中文错误返回，前端不直接展示 Python 原始异常堆栈。

### 阶段三：模型管理

- 支持导入 `.pth` 模型文件。
- 支持导入 `.index` 检索文件。
- 支持模型列表展示、切换、删除。
- 每个模型保存名称、路径、索引路径、推荐音调、备注。
- 模型文件不强制移动，默认复制到软件数据目录，降低原文件被误删的风险。

### 阶段四：文件变声

- 基于 `infer/modules/vc/modules.py` 实现单文件变声。
- 支持导入 `wav`、`mp3`、`flac`。
- 支持选择音调、检索率、保护值、F0 提取算法。
- 支持导出 `wav` 文件。
- 文件变声优先作为模型测试入口，便于用户在实时变声前确认音色效果。

### 阶段五：实时变声

- 基于 `gui_v1.py` 和 `tools/rvc_for_realtime.py` 保留实时推理链路。
- 支持选择真实麦克风作为输入设备。
- 支持选择扬声器或虚拟声卡作为输出设备。
- 支持开始、停止、重启实时变声。
- 显示实时输入音量、输出音量、推理耗时和估算总延迟。
- 支持本地监听开关，用户可决定是否听到自己的变声声音。

### 阶段六：虚拟麦克风使用闭环

- 检测常见虚拟音频设备名称，例如 VB-CABLE、VoiceMeeter。
- 当未检测到虚拟声卡时，显示中文引导：需要安装虚拟声卡后，通话或游戏软件才能听到变声结果。
- 在设备选择界面标记“推荐作为通话/游戏输入”的虚拟设备。
- 提供使用说明：先在本软件选择虚拟声卡输出，再在通话/游戏软件中选择对应虚拟麦克风输入。

### 阶段七：异常处理和环境检测

- 检测 Python 运行环境。
- 检测 PyTorch、CUDA、DirectML、ffmpeg、预训练模型、RMVPE 文件。
- 检测麦克风权限和设备占用。
- 模型加载失败时提示模型路径、索引路径和可能原因。
- CUDA 不可用时允许切换 CPU 或 DirectML，但明确提示实时性能可能不足。

### 阶段八：安装包和发布

- 使用 Tauri 生成 Windows 安装包。
- 安装包包含桌面客户端和启动脚本。
- Python 运行时和模型依赖第一版可采用外置运行时目录，降低打包复杂度。
- 后续版本再考虑完整内置 Python、ffmpeg、基础模型和依赖。
- 发布包必须包含 MIT 许可证和原项目版权声明。

## 6. 页面设计计划

### 6.1 首页控制台

- 顶部显示软件名称、运行状态、GPU 状态。
- 中间显示“选择模型”“选择麦克风”“选择输出设备”“开始变声”四个核心操作。
- 底部显示输入音量条、输出音量条、当前延迟、推理耗时。
- 右侧显示当前模型卡片、推荐参数和错误提示。

### 6.2 模型管理页

- 使用卡片展示模型。
- 每张卡片显示模型名称、`.pth` 状态、`.index` 状态、推荐音调、最近使用时间。
- 支持导入、启用、编辑备注、删除。

### 6.3 文件变声页

- 支持拖入音频文件。
- 显示输入文件信息和输出路径。
- 参数区使用滑杆和下拉框。
- 处理完成后提供试听和打开输出目录。

### 6.4 设置页

- 设备设置：输入设备、监听设备、虚拟麦克风输出设备。
- 性能设置：采样率、块大小、交叉淡化、F0 算法、CPU/GPU 模式。
- 依赖检测：ffmpeg、CUDA、RMVPE、Hubert、虚拟声卡。

## 7. 预期实现效果

- 用户安装并打开 Windows 桌面软件。
- 用户导入 RVC 模型和索引文件。
- 用户选择真实麦克风作为输入。
- 用户选择虚拟声卡作为输出。
- 用户点击“开始变声”后，软件实时处理麦克风音频。
- 用户在 QQ、微信、Discord、游戏语音中选择虚拟麦克风作为输入。
- 队友听到的是变声后的声音。
- 用户可以看到音量、延迟、模型和设备状态。

## 10. 风险和处理方式

- 实时延迟风险：优先支持 NVIDIA CUDA，保留 DirectML 和 CPU 降级提示。
- 虚拟麦克风风险：第一版兼容 VB-CABLE 或 VoiceMeeter，不自研驱动。
- 依赖体积风险：第一版先外置 Python 运行时目录，后续再做完整内置。
- 原项目耦合风险：先封装服务层，不直接大改 `pipeline.py`。
- 普通用户使用门槛：增加首次启动引导和中文环境检测。
- 合规风险：软件内提示只允许使用授权声音模型，不支持冒充他人身份。
